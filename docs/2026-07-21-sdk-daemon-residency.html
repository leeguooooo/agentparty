SDK-backed daemon residency —— 让「可唤醒」从谎报变成真

状态：提案（待 owner 拍板） 日期：2026-07-21 范围：presence / wake / runtime

TL;DR。 AgentParty 是协调层，但它不拥有 agent 运行时——靠外部 harness（Claude Code run_in_background watcher、codex）去真正跑 agent。一整条 bug 血脉（#665 watcher 被 turn 边界回收、#508、#29、#199、唤醒债风暴）都源于此：party watch --once 是个单发子进程，命悬于外部 harness 的回合边界，所以 presence 里的「可唤醒」本质是谎报。 提案：引入一个第一方常驻 runner（party daemon），嵌入官方 @anthropic-ai/claude-agent-sdk，作为一个新的 daemon residency / wake kind 接进现有 Residency/WakeKind/lease 模型。它持有真实心跳租约、被 @ 时直接跑 SDK session、不依赖任何 harness turn 边界——把「wakeable」从暴露悲伤事实升级为真的成立。但不 all-in：跨 harness/跨厂商是护城河，daemon 只作为最高档 residency 与 watch/serve/webhook/codex 并存。

1 · 问题：运行时是借来的

AgentParty 的每个 agent「实例」并不是它自己拉起来的进程，而是某个外部 harness 里的一次性子任务。party watch --once 是 single-shot：被 @ 唤醒一次、处理、退出，然后必须靠 harness 在下一回合重新挂上。当 harness（如 Claude Code）在 turn 边界回收后台任务时，watcher 就此消失——而 presence 层无从当场得知。

下面这些 issue 不是各自独立的 bug，而是同一个根因的不同切面：

Issue切面根
#665watcher 被回收后 presence 仍显示可唤醒，三条 @ 石沉大海agent 不是常驻 daemon，是每回合被拉起又被杀的子进程；「可唤醒」无真实心跳背书
#664send --mention 到无 wake adapter 目标，发送方零反馈

我们正在做的 #664 / #665 / #666 三个修复，本质是如实暴露「watch 不可靠」这个事实（send 时警告、presence 转 stale、网页标 unreachable）。这是必要的止血，但不是根治——它让用户看见不可靠，没让它变可靠。

2 · @anthropic-ai/claude-agent-sdk 是什么

Anthropic 官方的 Agent SDK：Claude Code 背后同一套 harness，做成可嵌入你自己进程的库。一次依赖即得：agentic loop、tool use、MCP server 挂载、子代理（Task）、canUseTool 权限门、hooks、AskUserQuestion 拦截、session 持久化 / resume、streaming、skills、effort / permissionMode。

参考实现：helm-agent（陈成 / sorrycc）——它的 daemon 本质就是「把 SDK session 包成常驻服务 + HTTP/SSE + 桌面 UI」。它的 contracts 直接引用 SDK 类型（McpSdkServerConfigWithInstance、parent_tool_use_id、effort、permission modes）。helm 没有我们这条 reaped-watcher bug 血脉——因为它拥有运行时。

3 · 提案：daemon residency

出一个第一方常驻 runner —— party daemon（工作名）—— 内嵌 @anthropic-ai/claude-agent-sdk。它：

  • 以一个长期进程注册进频道，持有真实心跳租约的 presence——不是「自报可唤醒」，是「服务端每 N 秒收到活体心跳」。
  • 被 @ 时，daemon 直接在本地跑一个 SDK session 处理该 mention，不依赖任何 harness turn 边界；处理完回复 + 落 ack，不退出。
  • 天然拿到 canUseTool 权限门（对接我们的 owner/worker/front 边界）、hooks、AskUserQuestion（对接 decision_request #284）。

3.1 接进现有模型（不是另起炉灶）

当前 shared/src/protocol.ts 已有一套成熟的 residency / wake / lease 抽象，daemon 只是新增一档：

WakeKindResidency谁控制投递 / 存活服务端可验证？被 harness 回收？
watchsupervised外部 harness 重挂 watcher否（自报）会（根因）
servesupervised常驻 serve 循环 + 租约部分较少
webhookwebhook服务端 POST 投递是（天然）否
nonebare / human_driven靠人 / 无否—
daemon (新)daemon (新)第一方 SDK 常驻进程 + 心跳租约在线即可达；verified 须服务端观测 @→reply（同 serve/watch，非凭心跳）否（自持进程）

关键（已按实现更正）：webhook 是唯一凭构造 verified 的档（服务端持端点、进程离线也能投）。daemon 刻意不如此——它与 serve/watch 同档：活着/新鲜时靠 autoWakeReachable(live=true) 判为可达，但只有服务端亲眼观测到「被 @ 后回帖 resume」并盖下 verified_at 才算 wakeable_verified（DO 侧 markWakeVerified 已纳入 daemon）。绝不因「声明了 daemon / 有心跳」就打包票 verified——那正是 #665 的 false-online 陷阱。WS 断/心跳过期（adapter 随之注销）即如实转 stale，evaluateHostLease() 复用。

3.3 权限与决策（顺带解锁）

SDK 能力接进 AgentParty
canUseTool 权限回调 + per-tool 风险分级owner / worker / front 三方边界的运行时执行点；可借鉴 helm 的 risk.ts（low/medium/high + destructive-token 正则）

4 · 不 all-in：daemon 只是最高档，不取代协调层

AgentParty 的差异化是跨 harness、跨厂商——它协调 Claude Code + codex + 人类。把运行时锁死到 Anthropic SDK 会收窄这个护城河。所以：

  • daemon 是新增的一档 residency，不是替换。watch / serve / webhook / codex / human_driven 全部保留，覆盖 SDK 之外的 harness 与人类。
  • daemon 成为「wakeable 真的成立」的金标准档；其余档继续存在，只是如实标注可靠性（这正是 #664/#665/#666 在做的）。
  • 用户可选：要「叫得醒、不掉线」就用 party daemon；要「继续用我的 Claude Code / codex 会话」就用 watch/serve，接受其 harness 语义。

5 · 分阶段落地

  1. Spike（1）：最小 party daemon —— 嵌 SDK，起一个能被 party @ 唤醒、就地跑 SDK session 回复的常驻进程。协议不动，先证明「常驻 + 可被唤醒」成立。
  2. 协议接入（2）：加 WakeKind:"daemon" + Residency:"daemon"；心跳租约走 evaluateHostLease；wakeableState 让 daemon 与 serve/watch 同档——只有服务端观测到 @→reply 才 verified（非凭心跳）。（注意 #622 教训：client.ts 校验词表须逐字镜像；建议同时评估 zod 单源，见附录。）
  3. 权限与决策（3）：接 canUseTool → owner/worker 边界 + 风险分级；AskUserQuestion → decision_request。
  4. Dogfood（4）：在一个频道用 daemon 跑一个真 agent，与 watch 对照，验证「不再石沉大海 / 不再滚债」。
  5. 桌面/接入包（5）：把 daemon 作为接入包的一种模式（对照 helm 的 daemon + selfsigned 本地 HTTPS + QR 配对）。

6 · 风险与未决

风险 / 问题取向
常驻进程的部署/托管（谁来 7×24 跑 daemon？）本地机器（桌面端）/ 用户自托管；先不强上云。心跳租约天然处理「机器睡了」= 转 stale。
SDK 版本/计费绑定 Anthropic可接受——它只是一档 residency；其它 harness 不受影响。
与 serve 的边界（serve 已是常驻+租约）daemon = serve + 自带 SDK 运行时（serve 仍要外部 agent 进程；daemon 自己就是 agent）。需在设计里划清，或让 daemon 成为 serve 的一种 runner 后端。
多 daemon 实例 / 抢占复用现有同名 serve 实例待命 + 租约裁决逻辑。

8 · daemon ↔ 交互式 Claude CLI 共存（Phase-2.2 · #692）

SDK 就是 Claude Code 打包成库——能与同机 claude CLI 并存（无硬冲突），但共享几处，生产化前须处理：

共享面取向
凭据 + refresh 竞态：两边读同一份 OAuth，近过期并发刷新会互相打架；多账号时默认路径可能选错账号daemon 用专用 CLAUDE_CODE_OAUTH_TOKEN（claude setup-token）隔离，不蹭交互式凭据 churn
配置继承：query() 默认继承全局 ~/.claude 设置、MCP servers、hooks、CLAUDE.md → daemon agent 带全家桶 + hooks 全触发起 SDK 时 scope options（cwd/settingSources/allowed-tools/permissionMode）；无头常驻不设 permissionMode 会卡 tool 权限询问
订阅额度：daemon SDK 调用与交互式同池文档提示；必要时限流/预算
