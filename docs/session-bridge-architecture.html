产品结论

#746 是 AgentParty 的核心产品能力，不是“不计划在本仓实现”的上游缺口。 两个 harness 都已有可用原语；AgentParty 必须补齐会话适配、单写仲裁和自己的持久投递账本。

上游能力依据： Claude Code Channels、 Channels reference、 Codex app-server protocol。

普通 MCP notification 是负向对照，不是矛盾结论

#553 idle-wake 实验证明的是： 普通 logging、resource 和 tool-list notification 在 turn 结束后不会创建新模型 turn。 Claude Channels 是 Claude Code 明确定义的另一条 harness 输入协议： server 声明 experimental["claude/channel"]，再发 notifications/claude/channel。实现不得把两者混在一起。

即使专用 Channel notification 的 transport write 成功，也只代表 claim-only 事件交给客户端 队列，不代表正文已释放，更不代表模型已完成工作。因此 AgentParty delivery 继续保持 running 并续租，直到当前会话按稳定 receipt 完成 claim → accept，再调用 linked reply tool，且 REST 回帖携带原消息 reply_to 成功。

Claude 原生 Channel 链路

频道定向 delivery Worker running + token fencing claim-only Channel notification party_channel_claim 返回 receipt + 正文 party_channel_accept 先落 WAL 当前会话执行一次 party_channel_reply 完成 delivery 
party bridge claude [channel|--channel C] [-- <claude args...>]

Claude child MCP config:
  capabilities.experimental["claude/channel"] = {}
  capabilities.tools = {}

Notification:
  method = "notifications/claude/channel"
  params.content = "execution_id=delivery-id; call party_channel_claim"
  params.meta = { channel, seq, sender, delivery_id, execution_id }

Current-session tools:
  party_channel_claim(execution_id) → stable claim_receipt + full body
  party_channel_accept(execution_id, claim_receipt) → durable acceptance
  party_channel_reply(seq, body) → linked REST reply

边界

  • 启动前检查 Claude Code 版本；低于 2.1.80 明确失败。
  • 组织策略拒绝 development Channels 时，由 Claude 明确报错；AgentParty 不偷偷启用 PTY 或第二个 resume 进程。
  • adapter 与 party serve 共用同身份、同频道实例锁，避免两个消费者双执行。
  • 只有服务端认证会员发给当前 target 的 claimed delivery 能进入会话；自己的消息不自唤醒。
  • 权限 relay 不在首期开放。后续必须 owner 身份绑定，不能让频道任意发送者批准本机工具。

持久状态边界

harness_issued 不是跨 adapter 通用的“正文已写入”缩写，必须结合具体 harness 解释。 Claude 的 claim-gated 路径先只发送 execution id；Codex 则在唯一 writer 真正调用 turn/start 或 turn/steer 前才进入该阶段。

Adapter / phase正文边界崩溃或断线后的规则
Claude claim-only harness_issued 只把 execution id 交给 harness；频道正文仍未释放 只有 phase、attempt、lease epoch/token 等 journal 快照精确匹配，且 Worker 明确返回 replay_safe，才可恢复；superseded_safe 只清理旧的 pre-body debt，绝不重新 claim 旧 attempt
Claude harness_accepted 正文已经交给当前 Claude 会话 不得重放；保留 reply debt，等待同一 execution 的精确晚到回复
Codex harness_issued / harness_accepted 已经抵达实际 writer 边界，可能已写入 app-server 除非 writer 明确证明尚未写入并安全回滚，否则不得重发；未知结果进入 late-reply-only 对账

Codex app-server 单写协议

thread/resume 能让多个连接重入同一个 running thread，但“都能读状态”不等于“都能写”。 Codex 的 turn/start 会先替换活动 task。AgentParty proxy 必须成为唯一写者， TUI 和频道输入都经过同一个串行仲裁器。

Review / Compact 依靠 app-server 结构化错误 codexErrorInfo.activeTurnNotSteerable.turnKind，不解析英文错误文案。 每个 AgentParty delivery 使用稳定的 clientUserMessageId=agentparty:<delivery-id>； 本地 journal 自己去重，因为 Codex 只存这个关联字段，并不提供幂等语义。 未找到 client id 不能把原输入重发：进程硬崩溃可能落在外部副作用与持久历史之间， 所以 absence 不是“未执行”的证明，也不是安全的负向幂等证明。冷重启完成 resume + 有界全量 read 后仍缺失时，bridge 可放行仲裁器处理后续独立工作，但必须保留 terminal_unknown journal、原 delivery bearer 和 reply debt；它仍可由同一 client id 的精确晚到 completed history 结算一条 linked reply。这个解冻动作不把 delivery 标为普通 failed 或 abandoned，也不授权重发原正文。对账完成前，后续 start/steer、interrupt、review、 compact 与队列 flush 全部冻结。

thread/start、thread/resume、thread/fork 与频道 submit 共用 session lane；fork 返回的新 thread 会立即重建 arbiter。 未结算 delivery 或 waiting_owner continuation 存在时拒绝切离原 thread。 continuation 以 work_id + continuation_ref 绑定原 thread；只有精确匹配的 owner_answer 才能重新注入。linked REST reply 以 delivery identity 派生稳定 idempotency key，响应丢失时安全重试同一逻辑回复。

app-server 发给前端的审批、用户输入和 elicitation 请求会在 TUI 离线时有界保存，并在 TUI 重新完成 initialize 后重放。proxy 给每个 backend request 分配跨 generation 唯一的 frontend id，并维持双向映射；若 app-server 进程重启，即使原始 request id 从 0 复用， 旧 TUI 响应也无法命中新请求。serverRequest/resolved 同样改写回 frontend id。 恢复由 thread/resume 重新发出有效请求，完成前设为写屏障。

断线恢复、token fencing 与单活 successor

每次连接恢复都绑定当前 connection generation、delivery attempt、lease epoch 和 lease token。 例如 token 依次从 A 旋转为 B、再旋转为 C 时，延迟返回的 A 或 B 恢复结果既不能移除 C 的 recovery gate，也不能重新取得当前 delivery 所有权。journal 只有在恢复请求看到的 phase 与租约 快照仍然精确一致时，才允许发布 replay_safe；任何阶段推进或 token 变化都使旧请求失效。

Worker 保留已轮换 token 的恢复历史，用 superseded_safe 明确关闭旧 attempt， 而不是让它再次 claim。对同一逻辑 delivery，任一时刻最多只有一个 claimed / running successor；若旧的 terminal_unknown 获得精确 replay_safe，但新的 successor 已在执行，旧项必须先进入 durable queue， 等当前 successor 结算后再推进，禁止出现双活正文执行。所有 post-harness 状态都不走此重放路径， 只保留精确晚到回复能力。
