AgentParty 桌面 UI 热更新设计

日期：2026-07-11 · 状态：实施中 · 适用：macOS、Windows、Linux 桌面壳

目标

React、CSS、文案等 UI 变更独立发布，不再要求每次生成 DMG/NSIS/AppImage。Rust、Tauri 权限、sidecar、系统插件和 UI ABI 变化仍走客户端版本发布。

信任边界：桌面壳只执行 AgentParty 官方签名的 UI 包。私有部署只提供 API 和认证配置，不能向本机下发可调用 Tauri IPC 的代码。

为什么不能直接加载远程站点

方案问题结论
WebView 直接打开部署 URL远端页面一旦被篡改，可调用 Keychain、更新器、本机 agent、通知和文件打开等 IPC。拒绝
私有服务器下发自己的 UI把每个私有部署都提升成桌面代码签名方，无法统一撤销和审计。首版不支持
官方签名 UI 包 + 本地协议可验签、缓存、回滚，并维持 Tauri 本地 origin 与 IPC 权限控制。采用

发布契约

{
  "schema": 1,
  "channel": "stable",
  "build_id": "<git sha>",
  "published_at": "2026-07-11T12:00:00Z",
  "ui_abi": 1,
  "min_shell_version": "0.2.94",
  "asset": { "url": ".../agentparty-ui-<sha>.tar.gz", "sha256": "...", "size": 123456 }
}

desktop-ui-latest.json 与 archive 分别签名。客户端固定访问官方 HTTPS endpoint，拒绝未知 schema、错误 channel、ABI 不匹配、壳版本过低、时间倒退、哈希或签名错误。

本地存储与启动

目录/状态责任
ui/bundled随应用携带的救援 UI，永不删除。
ui/builds/<build_id>验签并安全解包后的不可变 UI。
ui/state.json记录 current、previous、pending、最高已接受发布时间和失败次数。
agentparty-ui://localhost/只从 current 目录读取静态文件；规范化路径并禁止目录穿越。

下载先进入 staging；通过大小、文件数、路径、哈希、签名与兼容性校验后原子 rename。新包先标记 pending，React 完成初始化后调用 desktop_ui_ready(build_id, ui_abi)。连续两次未 ready 时回退 previous；previous 也失败则使用 bundled。

更新时机

应用启动后延迟检查，前台每 6 小时最多检查一次，并提供手动检查。下载完成后仅重载 WebView，不重启原生进程。正在编辑消息、执行配对或本机 agent 操作时延迟重载；强制安全撤回可在下次启动生效。

客户端版本边界

可热更新必须发客户端
React 页面、CSS、i18n、静态图片、仅调用既有 IPC 的交互Rust/Tauri、权限、capabilities、sidecar、协议处理器、签名公钥、UI ABI、窗口/托盘原生行为

验收

  1. 发布一个只改 UI 的提交，GitHub 生成签名 UI 包但不生成新 DMG。
  2. 已安装桌面版发现并加载新 build_id，设置和登录状态保留。
  3. 断网、签名损坏、archive 路径穿越、ABI 不匹配均保留当前 UI。
  4. 注入启动崩溃后自动回退 previous；删除缓存后仍能启动 bundled。
  5. 切换官方与私有 API origin 时使用同一份官方签名 UI。
