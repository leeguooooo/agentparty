桌面 UI 热更新实施计划

目标：官方签名 UI 独立发布，本地验签加载，启动失败自动回滚。

全局约束

  • 私有部署不能成为代码签名方。
  • ui_abi=1；首版最低壳版本 0.2.94。
  • 所有 archive 路径、大小、文件数、哈希和签名必须在解包前后校验。
  • 内置 UI 始终可启动；更新失败不能阻塞登录和频道。

文件与责任

范围文件产出
发布scripts/desktop-ui-release.ts、测试、.github/workflows/desktop-ui-release.yml确定性 archive、manifest、签名和固定 release channel。
原生核心desktop/src-tauri/src/ui_update.rs模型、兼容性、安全解包、状态机、原子切换。
原生接线desktop/src-tauri/src/lib.rs、tauri.conf.json固定 endpoint、后台检查、custom protocol、ready command。

恢复Rust 状态机测试 + 桌面 smokepending 两次失败回退 previous，再失败使用 bundled。

执行顺序

1. 发布通道

  1. 先写 manifest、非法元数据、确定性 archive 和 workflow 契约测试，并确认失败。
  2. 实现构建脚本；输出 versioned asset 与签名输入。
  3. 新增 main 分支 Web 变更触发的 workflow，使用 release environment 的 v2 signing secret。
  4. 运行脚本测试、TypeScript 和 workflow 静态测试。

2. 原生可测试核心

  1. 先写兼容性、路径穿越、symlink、超限、哈希错误和状态迁移测试。
  2. 实现纯 Rust manifest 与 cache API，不接网络和 WebView。
  3. 用临时目录验证 staging、atomic rename、current/previous 和清理策略。
  4. 运行 cargo test 与 cargo clippy -- -D warnings。

3. Web ready 握手

  1. 先写浏览器 no-op、合法注入值、重复调用 single-flight 与错误吞吐测试。
  2. 实现 reportDesktopUiReady()，只在 Tauri 且 build identity 完整时 invoke。
  3. 在 App 身份恢复和首屏初始化完成后调用；不等待返回，不阻塞 UI。
  4. 运行 Web 全量测试、类型检查和生产构建。

4. 原生接线

  1. 注册固定官方 endpoint 和 custom protocol，协议只读 current 或 bundled。
  2. 下载 manifest/signature/archive，调用已测试核心后切换 pending。
  3. 注册 ready command；超时/下次启动按状态机回退。
  4. 保留 frontendDist 作为 bundled 救援页。

5. 端到端门禁

  1. 发布测试 UI build，不提高客户端版本。
  2. 已安装应用更新 UI 并回报 build_id；切换私有服务器仍正常登录。
  3. 用损坏签名、路径穿越 archive、ABI 2 manifest 和 crash build 验证拒绝/回退。
  4. 通过后更新桌面文档，并把 UI build identity 显示到统一设置面板。

完成标准

只有独立 UI 发布、真实桌面加载、恶意包拒绝、自动回滚和 bundled 救援五项都有证据，才算完成。Developer ID 公证仍是客户端二进制发布的独立门禁。
